1. Recommended stack
Layer	Choice	Reason
Framework	Next.js + TypeScript	Best balance of simple, elegant, full-stack JS. App Router Route Handlers support REST endpoints inside the app directory.
UI	Tailwind CSS + shadcn/ui + Framer Motion	Minimal, modern, visually strong without becoming cluttered.
Auth	Auth.js	Supports secure auth patterns, database/JWT sessions, OAuth, and username/password stores.
Database	PostgreSQL	
ORM	Prisma	Simple schema, migrations, good TypeScript integration. Prisma Migrate keeps database schema changes versioned.
AI	OpenAI API	Use Structured Outputs for predictable JSON challenge/grading results. OpenAI states Structured Outputs enforce schema adherence better than JSON mode.
Hosting	Vercel	Still suitable. Vercel Hobby is free for personal projects, and GitHub deployments are supported.

Important hosting note: Vercel Hobby is for personal/non-commercial use, so once GURUnet adds payment, paid users, or commercial features, move to Pro or another paid deployment path. Vercel’s terms say Hobby is offered for personal/non-commercial use.

2. Product direction
GURUnet core idea

GURUnet is a personal engineering discipline platform for network engineering, cybersecurity, Linux, scripting, troubleshooting, automation, and documentation.

It should not feel like Hack The Box or TryHackMe visually. It should feel like a personal engineering command center:

Minimal dashboard
Daily challenge card
PIS/ERT/streak tracking
Submission and grading flow
Recovery penalties
Engineering notebook
Strict professor/senior-engineer correction tone
Future-ready social and payment system

5. Web-app behavior adapted from your task rules
Daily flow
At 08:00 local time, generate or unlock the daily challenge.
User sees:
title
difficulty
scenario
objective
constraints
allowed tools
expected answer format
deadline: 12:00 noon
User submits answer.
App records submission timestamp.
If the answer appears generic or unverifiable, app may trigger one follow-up verification question.
Grading is completed.
PIS, ERT, streak, tracker, and notebook are updated.
Solution is revealed only after submission/grading.
Missed challenge logic

If there is no submission after the allowed window:

status becomes Missed
score becomes 0/20
PIS decreases by -1.0
streak resets to 0
no ERT earned
next challenge includes a recovery component
Late restriction logic
Submission time	App effect
Before 12:00	No penalty
12:01–16:00	Late penalty applied
After 16:00	No PIS increase, no ERT
After 20:00 / next day	Grade allowed, but final score capped at 12/20
Access restriction ideas

These are good penalties for the web app:

Trigger	Restriction
Missed challenge	Next day includes recovery task
2 misses/week	Weekly PIS gain capped by 50%
3 misses/week	Weekend recovery challenge required
Low score under 13/20	No ERT, no PIS gain
Unsafe answer	Score cap and temporary “Safety Review” badge
Repeated vague answers	Harder verification questions
Late after 16:00	No rewards even if answer is good
Unredeemed poor discipline	Dashboard shows “Recovery Required” state

This makes the app feel serious without becoming hostile.

8. AI implementation strategy

Use AI only for:

challenge generation
grading assistance
verification question generation
notebook summary generation

Do not let AI directly change PIS/ERT balances. The app logic should compute that deterministically.

Use OpenAI Structured Outputs for challenge and grade JSON, because schema adherence is important for storing results safely.

For token control:

keep the grading rubric static and cached
store challenge templates
send only the relevant challenge + submission + minimal historical weakness summary
use cheaper models for notebook summaries
use stronger models only for grading complex submissions
log token usage per request

OpenAI prompt caching works automatically for recent models and can reduce latency and input token cost when prompts share the same prefix; static instructions should be placed at the beginning and variable user content at the end.

12. Challenge generation logic

The challenge generator should receive:

user PIS
recent weaknesses
missed status
weekly discipline record
topic rotation history
difficulty level

It should output structured JSON:

{
  "title": "...",
  "difficulty": "Normal",
  "topic": "OSPF Troubleshooting",
  "scenario": "...",
  "objective": "...",
  "constraints": ["..."],
  "allowedTools": ["..."],
  "expectedAnswerFormat": "...",
  "submissionDeadline": "12:00",
  "solution": "...",
  "antiGenericRequirement": "Explain why your first three checks are ordered that way."
}

The solution must be stored but hidden until submission/grading.

13. Grading logic

The AI may propose scores, but the backend must apply deterministic caps:

rawScore = ((creativity + ingenuity + reporting + alienness + neatness) / 35) * 20

finalScore = rawScore
           - balancePenalty
           - latePenalty

then apply:
technical correctness cap
unsafe cap
incomplete cap
after-20:00 cap

The app should never trust the AI alone for:

missed status
late penalty
ERT balance
PIS balance
streak count
redemption deduction

Those must be handled by code.

14. ERT redemption behavior

User enters:

Reward name
Cost
Date
Note

Backend checks:

if cost <= currentBalance:
    deduct
    log redemption
else:
    reject

Never allow negative ERT.

15. Deployment guideline for later

When ready:

Push project to GitHub.
Import repo into Vercel.
Add environment variables in Vercel.
Connect managed PostgreSQL database.
Run Prisma migration during deploy.
Set production domain.

Vercel supports Git-based deployments where commits or pull requests trigger deployments, and GitHub projects can be automatically deployed with preview URLs.

16. Build order

Do not build everything at once. Build in this order:

Order	Build item	Reason
1	Auth + protected dashboard	Foundation
2	Prisma schema	Data foundation
3	Manual challenge creation	Avoid AI complexity first
4	Submission flow	Core user action
5	Grading engine	Core value
6	PIS/ERT engine	Discipline/reward loop
7	Missed/late/recovery logic	Penalty loop
8	AI challenge generation	Automation
9	Notebook	Knowledge retention
10	Hosting	Only after local app works
